2. Security by Design — intégration par phase projet

La sécurité est intégrée à chaque phase du projet, pas uniquement en fin de cycle. Chaque action de sécurité est associée à un responsable et à une échéance.

PhaseSprintsActions sécuritéResponsable
Cadrage S1–S2 Analyse des risques sécurité · Classification des données · Définition des exigences sécurité du projet · Validation de l'architecture par le RSSI RSSI · CDP
Architecture S3 DAT soumis à revue RSSI · Choix Azure WAF + Key Vault · Configuration réseau (VNet, NSG) · Activation Azure Defender · TLS 1.2+ forcé Lead Dev · RSSI
Développement S3–S9 OWASP intégré dans la DoD · Revue de code sécurité sur chaque PR (Lead Dev) · Scan SonarQube à chaque commit · Aucun secret en clair dans le code (Key Vault) Lead Dev · Dev / ESN
Intégrations S5–S8 Authentification OAuth2 / JWT pour les APIs · MFA Azure AD B2C activé (J4) · Chiffrement des échanges CRM et paiement · Validation des tokens côté serveur Lead Dev · RSSI
Qualification S9–S10 Tests OWASP ZAP (automatisés) · Pentest par prestataire externe · Revue des résultats avec RSSI · Plan de remédiation des failles identifiées RSSI · Prestataire
Go-live S11–S12 Validation finale RSSI (0 faille critique) · Azure Monitor + alertes sécurité activés · Plan de réponse aux incidents documenté · PV sécurité joint au PV de recette RSSI · DSI
3. Contrôles techniques retenus
🔐 Authentification & Accès
✓
MFA obligatoire pour tous les utilisateurs (Azure AD B2C)
✓
RBAC (Role-Based Access Control) — droits minimaux par rôle
✓
Sessions JWT avec expiration courte (15 min + refresh token)
✓
Comptes de service DSI séparés des comptes utilisateurs
✓
Aucun compte générique ou partagé en production
☁️ Infrastructure Azure
✓
Azure WAF — filtrage des requêtes malveillantes (OWASP ruleset)
✓
Azure Key Vault — stockage chiffré de tous les secrets et certificats
✓
HTTPS / TLS 1.2 minimum forcé (TLS 1.0 / 1.1 désactivés)
✓
Azure Defender for App Service — détection des anomalies
✓
Sauvegardes PostgreSQL chiffrées — rétention 30 jours
🛡️ Sécurité applicative
✓
Headers HTTP sécurisés (CSP, HSTS, X-Frame-Options, X-Content-Type)
✓
Validation des entrées côté serveur (pas uniquement côté client)
✓
Requêtes SQL paramétrées (ORM — 0 concaténation de chaîne)
✓
Protection CSRF sur tous les formulaires (tokens)
✓
Données sensibles jamais dans les logs ni dans les URLs
📊 Surveillance & Réponse
✓
Azure Monitor — alertes temps réel sur anomalies de trafic
✓
Logs d'accès aux données sensibles centralisés et horodatés
✓
Alertes RSSI automatiques sur vulnérabilité critique détectée
✓
Plan de réponse aux incidents documenté avant go-live
✓
Plan de rollback testé et validé (retour version précédente < 1 h)
4. Couverture OWASP Top 10 — mesures appliquées
RangRisque OWASPMesure retenueCouverture
A01 Broken Access Control RBAC Azure AD B2C · Vérification des droits côté serveur à chaque requête · Tests Playwright sur les accès non autorisés Dev + Infra
A02 Cryptographic Failures TLS 1.2+ obligatoire · Données au repos chiffrées (Azure PostgreSQL) · Secrets dans Key Vault uniquement Infra Azure
A03 Injection (SQL, XSS, CSRF) ORM paramétré (0 concaténation) · Validation des entrées · Tokens CSRF · Encodage des sorties HTML · Scan OWASP ZAP Dev + Tests
A04 Insecure Design Security by Design dès le cadrage · Revue architecture RSSI (J2) · Threat modeling réalisé en S3 Architecture
A05 Security Misconfiguration IaC (Infrastructure as Code) — configuration Azure versionnée · Headers HTTP sécurisés · Environnements dev/prod strictement séparés Infra Azure
A06 Vulnerable & Outdated Components OWASP Dependency Check dans le pipeline CI/CD · Mise à jour des dépendances obligatoire avant chaque sprint review CI/CD
A07 Identification & Auth Failures MFA Azure AD B2C · JWT à expiration courte · Blocage après 5 tentatives échouées · Pas de mot de passe en clair Infra Azure
A08 Software & Data Integrity Failures Signature des artefacts de déploiement · Vérification intégrité dans le pipeline CI/CD · Dépôt Git protégé (branch protection) CI/CD
A09 Security Logging & Monitoring Failures Azure Monitor + logs centralisés · Alertes automatiques RSSI · Logs d'accès aux données sensibles horodatés et non modifiables Infra Azure
A10 Server-Side Request Forgery (SSRF) Validation stricte des URLs côté serveur · Liste blanche des domaines autorisés pour les appels externes · WAF Azure configuré Dev + WAF
5. Gestion des vulnérabilités — SLA de remédiation

Toute vulnérabilité identifiée (scan SonarQube, OWASP ZAP, ou pentest) déclenche une procédure de remédiation avec un délai contractuel selon sa sévérité.

CRITIQUE (CVSS ≥ 9)
24 h
Correctif obligatoire · Escalade Sponsor · Go-live bloqué si non résolu
ÉLEVÉE (CVSS 7–8,9)
72 h
Correctif dans le sprint courant · Alerte RSSI + CDP · Suivi Copil
MOYENNE (CVSS 4–6,9)
Sprint suivant
Planifié dans le backlog · Suivi hebdomadaire · Toléré au go-live si plan actif
FAIBLE (CVSS < 4)
Fin de projet
Enregistré dans le backlog · Traité en amélioration continue post go-live
