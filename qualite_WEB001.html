1. Méthode TDD — cycle Red / Green / Refactor
ÉTAPE 1
RED
Écrire le test AVANT le code. Le test échoue car la fonctionnalité n'existe pas encore.
→
ÉTAPE 2
GREEN
Écrire le minimum de code pour faire passer le test. Pas d'optimisation encore.
→
ÉTAPE 3
REFACTOR
Améliorer et nettoyer le code sans casser les tests. Réduire la dette technique.
→
BOUCLE
REPEAT
Recommencer pour chaque nouvelle fonctionnalité tout au long des sprints.
Pourquoi TDD pour Time'Eats ?
  • › 2 ESN externes → garantir la qualité du code livré
  • › Périmètre ambitieux → éviter les régressions entre sprints
  • › Migration de données sensible → zéro bug en production
  • › Pipeline CI/CD → filet de sécurité automatique
TDD + Scrum : articulation
  • › Chaque user story a ses tests écrits AVANT le dev
  • › Les tests font partie de la Definition of Done
  • › CI/CD exécute les tests à chaque commit (GitHub Actions)
  • › La sprint review inclut un rapport de couverture SonarQube
Outils TDD retenus
  • › Jest — tests unitaires JavaScript / React
  • › React Testing Library — tests composants UI
  • › Playwright — tests End-to-End (E2E)
  • › SonarQube — qualité de code + couverture
  • › Postman / Newman — tests API

3. Definition of Done — critères d'acceptation des livrables

Une user story n'est considérée terminée que si TOUS ces critères sont satisfaits. Validée en sprint review par le PO.

✓
Tests unitaires écrits AVANT le code (TDD)
✓
Couverture de tests ≥ 80 % sur le module livré
✓
Tous les tests passent (0 test en échec)
✓
Tests d'intégration API validés
✓
Aucune régression sur les modules existants
✓
SonarQube : 0 bug bloquant · 0 vulnérabilité critique

Pas de code dupliqué (DRY respecté)
